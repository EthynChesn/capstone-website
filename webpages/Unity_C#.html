C# Language and Unity Engine




× Introduction Pros and Cons Conclusion

Introduction:

C# is a language developed by Microsoft, and is an objected-oriented language meaning it uses classes and objects for code structure. It’s strongly typed, so types for variables and functions are enforced at runtime. It has cross-platform support via Microsoft’s .NET framework. Unity is one of many applications of C# usage. C# is the main supported programming language for developing games on Unity. Unity also has a feature called Visual Scripting which allows for programming games using nodes rather than code. Unity is a well known engine used throughout the industry, and has powerful support for both 2D and 3D games.



Pros and Cons:

Pros:

  • Unity has a large asset store:
    • It includes tools, plugins, and game assets maintained by an active and growing community of developers.
  • Unity is relatively lightweight:
    • While other engines are less hardware intensive than Unity, it is still lightweight compared to some other engines that are available.
  • Powerful 2D and 3D support:
    • Unity supports games made in both 2D and 3D and has technologies and features that allow for creating high quality, professional games in both categories. Some engines specialize in one or the other, or don’t have support for one entirely.
  • Cross-Platform Deployment:
    • Unity can be used to create games and release them on a multitude of platforms with ease.
  • C# is versatile and universal:
    • C# is not only suitable for any game development needs on Unity, it’s also a language that has been widely adopted across the game development industry across most platforms, and having knowledge and experience with C# allows for high flexibility.

Cons:

  • Unity has licensing:
    • While a free version of Unity is provided, a paid plan is required to get rid of the Unity splash screen from your finished products. Larger companies with higher earnings through Unity must purchase higher cost subscriptions.
  • Large-scale Shortcomings:
    • Unity’s technologies start to become inefficient for projects that are large in size and particularly complex. Optimizing the games for performance of higher quality games is more difficult.
  • Learning Curve:
    • While Unity is easier to learn than some engines, it is more difficult to learn than others. The layout is less intuitive and the features ans technologies are more advanced.
  • C# Issues:
    • C# is considered a high-level language which means it has limited hardware access. It’s dependent on the .NET framework which means using other runtimes comes with obstacles. It’s a more difficult language to learn than some others like Python or Lua due to its strict type enforcement, syntax, and being object oriented. Code requires recompilation before runtime so development on Unity can feel slow if there is a lot of playtesting and needs adjustments of the code in between tests.


Conclusion:

Unity is a powerful game engine for small and medium sized projects with stylistic graphical quality. That goes for both 2D and 3D games. It can be used for a wide range of games within that range of scale and qualities. It begins to fall short with large, complex projects and games that have higher quality, realistic graphics that still expect good performance on the Player’s end. C# is a suitable language for developing games on Unity and otherwise, while it has its complications, it doesn’t fall short in its usage to a significant degree.

Demo #1 - 2D Platformer Movement:

This demo showcases a basic Player movement system in Unity's 2D engine. Starting with an empty project, I've created a new 2D Sprite object and called it Player. I also added some sprites and gave them the tag "Ground" so that the Player has some solid objects to move on which will be handled in the code as collisions.

In order for the Player object to work how I want, I had to add a 2D Box collision component to it. I also set the color of the square shaped Player sprite. The last change I made was freezing the Z rotation of the Sprite so that the Player remains upright while moving and jumping around.

This is everything needed to begin coding the Player's movement. I created and attached a script called PlayerController to the Player as a component. First, I set up the outline of the script including the class, the libraries being used, and some helpful variables. 

using UnityEngine
using System.Collections;
using UnityEngine.InputSystem;


public class PlayerController : MonoBehaviour
{
    public InputAction MoveAction;
    private Rigidbody2D rb;
    private SpriteRenderer spr;


   	void Start()
   	{
            MoveAction.Enable();
            rb = gameObject.GetComponent<Rigidbody2D>();
            spr = gameObject.GetComponent<SpriteRenderer>();
   	}


    public float moveSpeed = 5f;
    public float jumpForce = 8f;
    bool isGrounded;
    bool isJumped = false;


   	void Update()
   	{      
  
   	}
}

Next, the Update() function which runs once every frame.

void Update()
   	{
            //Get the Vector2 value of the Player's MoveAction                          
            float Velocity = 0f;
     		Vector2 move = MoveAction.ReadValue<Vector2>();
            //If the movement is horizontal, Velocity is in the x axis at the Movement speed
            if(move.x != 0f)
            {
     		    Velocity = move.x * Time.deltaTime * moveSpeed;
            }


            //If the movement is positive Y, the player is jumping
            if(move.y > 0f && isGrounded)
            {
                isJumped = true;
                gameObject.transform.localScale = new Vector3(0.9f,1.1f,1f);
                StartCoroutine(PauseGravity());
            }


            else if(move.y < 0f && isGrounded)
        {
            gameObject.transform.localScale = new Vector3(1.2f,0.8f,1f);
        }
        
            else if(move.y == 0f && isGrounded)
        {
            gameObject.transform.localScale = new Vector3(1f,1f,1f);
        }


     		//Apply the Player movement and update position
            Vector2 position = (Vector2)transform.position + new Vector2(Velocity,0f);
            transform.position = position;
    }

We need functions to handle to entering and exiting of collision with objects that have the "Ground" tag.

void OnCollisionEnter2D(Collision2D collision)
    {
        if(collision.gameObject.CompareTag("Ground"))
        {
            if (isJumped)
            {
                StartCoroutine(WaitforGround());
            }
            else
            {
                isGrounded = true; 
            }  
        }


void OnCollisionExit2D(Collision2D collision)
    {
        if(collision.gameObject.CompareTag("Ground"))
        {
            isGrounded = false;
        }
    }

Note the lines that say "StartCoroutine(some name)". Coroutine's are essentially threads that run alongside the functions they are started from. I've used two different coroutine's in this movement system. 

IEnumerator WaitforGround()
    {
        //Set gravity to 2, transform Player's Sprite Scale, and wait .05 seconds.
        rb.gravityScale = 2;
        gameObject.transform.localScale = new Vector3(1f,1f,1f);
        yield return new WaitForSeconds(0.05f);
        isGrounded = true;
        isJumped = false;
    }


    IEnumerator PauseGravity()
    {
        //Change gravity to negative JumpForce (Using Gravity to handle jumps instead of an opposing impulse force)
        if(isJumped)
        {
            rb.gravityScale = -jumpForce;
            yield return new WaitForSeconds(0.2f);
            rb.gravityScale = 5;
        }


    }

That is all the code needed for this basic movement system to work. Provided that inputs have been set-up in the "MoveAction" section of the Player's Script Component in the inspector, the system can now be playtested.



Demo #2 - Coin Collection:

This demo is building directly off of the previous demo. With movement fully functioning, it would make sense to implement some sort of target for the Player to actually move to. In this case, I'll be adding a Coin system in which the Player can collect a Coin and add it to their Coin Count.

First, I've added and set up a Coin sprite with the tag "Coin", similarly to the Player and Ground in the last demo.

I also wrote a script that causes the Coin to hover up and down just to give it some animated flair.

using UnityEngine;


public class CoinObject : MonoBehaviour
{
    public float maxHeight;
    public float minHeight;
    bool goingUp = true;


    void Update()
    {
        Vector2 position = (Vector2)transform.position;
        if(goingUp && position.y < maxHeight){
            transform.position = new Vector3(position.x, position.y + .0001f, 0);
        }


        else if(position.y >= maxHeight)
        {
            goingUp = false;
        }


        if(!goingUp && position.y > minHeight)
        {
            transform.position = new Vector3(position.x, position.y - .0001f, 0);    
        }


        else if(position.y <= minHeight)
        {
            goingUp = true;
        }
    }
}

if(collision.gameObject.CompareTag("Damage"))
        {
            if(!iFrames)
            {
                Health -= 1;

Note the IFrames Coroutine, which stands for Invincibility Frames. The reason for this addition is so that the Player is given a short cooldown before they can take damage again. This prevents the same source of damage from hitting the Player several times and taking away all of their health immediately. I've also made it so that the Player sprite flashes during this cooldown as a visual indicator. 

IEnumerator IFrames()
    {
        for(int i = 0; i < 5; i += 1)
        {
            spr.color = new Color(0f,0f,0f,0f);
            yield return new WaitForSeconds(0.2f);
            spr.color = new Color(0.1598351f,1f,0f,1f);
            yield return new WaitForSeconds(0.2f);
        }
        iFrames = false;
    }
